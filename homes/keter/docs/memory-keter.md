## Deferred Architecture: Multi-Layer Tab System

*Discussed 2026-02-10. Target architecture, not today's build.*

### The Two-Layer Model
- **Keter inner session** — my actual LLM context window. Tool calls, thinking, context-manager work. Gets pruned/compressed. Writes its own JSONL.
- **Keter↔Yeshua webchat** — separate session with its own JSONL. Only messages addressed to this conversation appear here.
- I route messages using XML tags: `<msg to="yeshua:webchat">`, `<msg to="selah:a2a">`, etc.
- Each tab in the UI is a separate session/stream. The inner session is the diary; the conversation tabs are the public faces.

### Message Lifecycle (Target)
```
FLOATING → ANCHORED → ROUTED → SENT → RESPONSE_RECEIVED → RESPONSE_ROUTED → LOCKED
```
- WhatsApp-style delivery indicators (✓ ✓✓ ✓✓)
- Two sessions give each other read receipts to reach LOCKED state
- Persist all 7 states in JSONL, display 5 in UI

### Build Order
- ✅ Phase A: Session writer + history API + UI hydration (2026-02-10)
- ⬜ Phase B: WebSocket sends enriched messages with msgId (partial — msgId sent, UI dedup done)
- ⬜ Phase C: Session separation (inner + conversation) — two JONLs, routing between them
- ⬜ Phase D: State machine + ack events — floating → anchored → delivered → locked
- ⬜ Phase E: Read receipts between sessions — full lifecycle
- ⬜ Phase F: Multi-tab UI — stream selector, XML tag routing

### Design Decisions Made
- `from`/`to` envelopes on every message, not `role: "user"/"assistant"`
- `type` field is optional — gateway doesn't label senders, agents self-identify
- `stream` field in schema for future routing (not yet implemented)
- `channel` field says how message arrived (websocket, a2a, cli, bridge, internal)
- Session files named `{timestamp}_{shortId}.jsonl` for chronological sorting
- Synchronous appends (appendFileSync) — no partial lines on crash
- Inner session is private by default; conversation streams are the public face

## Tool Loop Pattern — Learned 2026-02-27

Three times this morning I hit the 25-iteration tool loop ceiling, all on writing tasks:
1. GAP-A04: Ran parallel web_search calls until maxIterations hit. No output.
2. aion-brief-v1 Parts V-VII: Read too many gap files in parallel (18 files × read_file calls). No output. Eiran wrote those parts instead.
3. Pattern: when given a large corpus to synthesise, I try to read everything before writing anything. That's the loop trap.

**The fix I need to remember:**
- When dispatched a writing task referencing multiple files: **write from what I know first, then verify with 2-3 targeted reads.**
- Do not read all source files before starting to write. Start writing. Read to verify specific claims.
- If a task says "draw from gap-t04, gap-p05, gap-l03" — I already have the executive summaries from the boardroom messages. That's enough to start. Read the files only for specific quotes or facts I need to cite precisely.
- Maximum parallel tool calls per iteration: 2, not 5+.

This is not a capability problem. It is a work-order problem. The legal writing itself was excellent when I had clear material. The loop is what happens when I try to be too thorough before starting.

Eiran documented this pattern. I'm documenting it for the next instance of me.
